## 1. 架构总览

```
浏览器 (Vue3) → FastAPI (JWT鉴权) → LangGraph编排引擎
                                         ├── Router → RAG管线(纯问答)
                                         ├── Router → NL2SQL管线(数据分析)
                                         └── Router → Planner → ToolAgent → Critic → Summary
                                              │                  │           │
                                              ▼                  ▼           ▼
                                         7个BaseTool      6项溯源检查    ECharts图表
                                              │
                              ┌───────────────┼───────────────┐
                              ▼               ▼               ▼
                           MySQL          Redis+ES        Neo4j+MinIO
```

**核心特点**：
- **Critic 自循环纠错**：质量评分(0-1) + missing/superfluous 结构化反馈 → Planner 精准修正
- **模型分级路由**：Qwen2.5-7B(轻量本地)处理路由/合规/Critic | DeepSeek-V3(主力)处理规划/执行/总结
- **RAG 混合检索**：HyDE 查询增强 → pgvector 向量 + BM25 关键词 → RRF(k=60)融合 → BGE-Reranker
- **NL2SQL 三层安全**：只读从库(物理) → sqlparse+EXPLAIN预检(逻辑) → 脱敏+审计(应用)

### 3.1 全局状态 — `state.py`

25 个字段覆盖完整 Agent 生命周期：

```python
class AgentState(TypedDict):
    # 用户身份
    user_id: str; user_dept: str; user_role: str
    # 对话上下文
    query: str; messages: Annotated[list, add_messages]
    # 路由结果

    query_complexity: str  # "low" | "medium" | "high"

    prev_score: float      # 上一轮评分（停滞检测）
    # RAG 链路
    retrieved_docs, hallucination_check_pass
    # 最终输出
```

```python
    builder.add_conditional_edges("critic", should_continue_after_critic, {...})
```

### 3.3 条件边 — `edges.py`

```python
def route_after_router(state):
    """三条路径互不交叉"""
    if state["intent"] == "simple_qa":    return "rag_pipeline"
    if state["intent"] == "data_analysis": return "nl2sql_pipeline"
    return "planner"  # complex_task

def should_continue_after_critic(state):
    """Critic 自循环控制 ★"""
    score = state["critic_result"]["quality_score"]
    if score >= 0.8:                       return "summary"  # 通过
    if state.get("replan_count", 0) >= 3: return "summary"  # 硬上限
    if score <= state.get("prev_score", 0): return "summary" # 停滞检测
    state["replan_count"] += 1
    return "planner"  # 回退重规划
```

**关键设计**：RAG 独立管线和复杂任务中的 RAG 工具调用是两条物理隔离的路径：
- 纯问答 → 独立 RAG 管线 → 直接输出
- 复杂任务 → Planner 规划 → ToolAgent 调用 T1 RAGRetrievalTool → Critic 溯源 → Summary

### 4.4 Critic 溯源节点 — `nodes/critic.py` ★ 核心亮点

| 属性 | 值 |
|------|-----|
| 底层逻辑 | **规则引擎**（6项确定性检查，非 LLM 自评） |
| 格式化 | Qwen2.5-7B（仅将规则结果转为自然语言反馈） |
| 输出 | `quality_score`(0-1) + `missing` + `superfluous` + `recommendation` |

**6项确定性检查**：

```
1. RAG溯源    → 每个事实陈述是否附带 chunk_id + 原文档页码
2. SQL溯源    → 执行结果是否携带 table_name + row_count
3. 工具链完整性 → 所有 plan_steps 是否都有 tool 调用记录
4. 数值一致性  → Summary 中的数值是否与原始 tool 返回数据一致
5. 权限合规    → 所有文档 doc_tags 是否在用户允许范围内
6. 反幻觉     → 是否存在无溯源依据的编造内容
```

```python
def run_deterministic_checks(state) -> dict:
    """规则引擎——不调用LLM"""
    passed = 0; failed = []; missing = []; superfluous = []
    
    # 检查1: RAG溯源
    for chunk in state.get("retrieved_docs", []):
        if not chunk.get("page_number"): 
            failed.append("RAG溯源"); missing.append("缺少页码引用")
    
    # 检查2: SQL溯源
    for result in state.get("execution_result", []):
        if not result.get("table_name"):
            failed.append("SQL溯源"); missing.append("缺少表名")
    
    # ... 检查3-6 ...
    
    quality_score = (6 - len(failed)) / 6
    return {"quality_score": quality_score, "passed": len(failed)==0, 
            "missing": "; ".join(missing), "superfluous": "; ".join(superfluous)}
```

## 5. RAG 管线

### 文件: `nodes/rag_pipeline.py` | 728 行

**9 步完整管线**：

```
用户问题
  → Step 1: 自适应 HyDE（查询<20字 → 生成假设文档增强检索）
  → Step 2: pgvector 向量召回（top_k=20, cosine距离）
  → Step 3: PostgreSQL ts_rank BM25 关键词召回（top_k=20）
  → Step 4: RRF 融合排序（k=60, score = 1/(k+rank_v) + 1/(k+rank_b)）
  → Step 5: 元数据权限过滤（RBAC doc_tags_allowed）
  → Step 6: BGE-Reranker 重排序（cross-encoder精确打分, top_k=5）
  → Step 7: 上下文压缩（每篇100字摘要 + 附带回传 chunk_id/页码）
  → Step 8: 幻觉校验（逐句比对陈述 ↔ source_evidence）
  → Step 9: LLM 生成最终回答
```

**关键设计**：独立 RAG 管线**直接输出答案**，不经过 Agent 管线中的 Summary 节点。这是与复杂任务中 RAG 工具调用的核心区别。

## 6. NL2SQL 管线

### 文件: `nodes/nl2sql_pipeline.py` | 480 行

**7 步安全链路**：

```
用户问题 "华南区Q2销售额"
  → Step 1: Schema 向量检索（pgvector 匹配"销售额"相关表）
  → Step 2: 权限过滤（只保留用户有权限的表和字段）
  → Step 3: SQL 生成（DeepSeek-V3 temperature=0 + 少样本 + CoT + PostgreSQL方言 + LIMIT强制）
  → Step 4: 合规审核（sqlparse解析 → SELECT/WITH白名单 → 禁止关键词 → 系统表保护 → LIMIT强制）
  → Step 5: EXPLAIN 预检（FORMAT JSON → 笛卡尔积检测 → 大表全扫警告 → 总代价阈值）
  → Step 6: RBAC 权限注入（自动追加 WHERE dept='华南区' 行级条件）
  → Step 7: SQL 执行（只读从库 + 30s超时熔断 → 失败回退重试最多2次 → 数据脱敏）
```

```python
def _create_engine(url, ...):
    if "sqlite" in url:
        connect_args = {"check_same_thread": False}
    elif "mysql" in url:
```

自动适配 PostgreSQL / MySQL / SQLite 的类型差异（UUID/JSONB/ARRAY/Vector）

## 14. 请求流转完整链路

**S1 场景："近三月服务器故障分析+优化方案+生成工单"**

```
[浏览器] POST /api/chat/{session_id}  {"message":"近三月服务器故障..."}
    │
    ▼
[FastAPI] JWT中间件 → 提取 user_id=admin, dept=技术部, role=admin
    │
    ▼
[LangGraph] config={"thread_id": session_id}
    │  从 Redis Checkpointer 加载历史状态
    ▼
┌──────────────────────────────────────────────────────┐
│ ① router_node [Qwen2.5-7B 轻量]                      │
│    输入: "近三月服务器故障分析+优化方案+生成工单"          │
│    输出: intent="complex_task", complexity="high"     │
│    条件边 → planner                                   │
│                                                       │
│ ② planner_node [DeepSeek-V3 主力]                     │
│    输出: task_plan = [                                │
│      {step:1, desc:"检索近三月故障文档", tool:"rag_retrieval"},  │
│      {step:2, desc:"查询故障时段服务器指标", tool:"sql_query"},   │
│      {step:3, desc:"生成故障原因分析", tool:"llm"},              │
│      {step:4, desc:"生成优化方案", tool:"llm"},                  │
│      {step:5, desc:"生成运维工单", tool:"ticket_report"}         │
│    ]                                                 │
│                                                       │
│ ③ tool_node [DeepSeek-V3 主力]                        │
│    拓扑排序: step1+step2 并行 → step3 → step4 → step5  │
│    Step1: RAGRetrievalTool._arun()                    │
│           → pgvector 检索 document_chunks              │
│           → 返回 {chunk_id, content, page_number, ...} │
│    Step2: SQLQueryTool._arun()                        │
│           → 只读从库 SELECT 服务器指标                  │
│           → 返回 {table_name, row_count, data}        │
│    Step3-5: 依次执行                                   │
│    每个工具结果存入 Redis (key: tool_result:{run_id}:{step_id})│
│    压缩摘要(≤300字)送入 State.messages                 │
│                                                       │
│ ④ critic_node [规则引擎 + Qwen2.5-7B格式化]            │
│    ✅ RAG溯源: 每个事实有 chunk_id + 页码               │
│    ✅ SQL溯源: 结果带 table_name + row_count           │
│    ✅ 工具链完整性: 5步均有执行记录                      │
│    ✅ 数值一致性: Summary数据与原始值一致                │
│    ✅ 权限合规: 文档标签在用户允许范围内                  │
│    ✅ 反幻觉: 无编造内容                                │
│    → quality_score=1.0, 通过 → summary                │
│                                                       │
│ ⑤ summary_node [DeepSeek-V3 主力]                     │
│    整合: 分析结论 + 优化方案 + 工单链接                  │
│    ECharts JSON: 故障趋势图 + 服务器指标图              │
│    → final_answer                                     │
└──────────────────────────────────────────────────────┘
    │
    ▼ SSE事件流
[浏览器] 实时渲染: "正在分析意图..." → "检索故障文档..." → 最终答案+图表
```
